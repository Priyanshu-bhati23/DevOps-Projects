What does

01

Serves a REST API

A Flask app exposes /health, /count, and /reset endpoints — a real web service you can curl or build a frontend on top of.

02

Persists data in Redis

Every POST to /count atomically increments a counter stored in Redis. Data survives container restarts via a named Docker volume.

03

Runs in containers

Everything — app, database, monitoring — runs in Docker containers. One command boots the entire stack on any machine. No "works on my machine".

04

Ships automatically

Every push to main triggers GitHub Actions: lint → test → build Docker image → push to Docker Hub. Zero manual steps.

05

Observability built-in

Prometheus scrapes live metrics from the app every 15 seconds. Grafana visualises request rates, latencies, and error counts in real time.

layer 1 — application

The Flask app

app/main.py
import os, redis
from flask import Flask, jsonify
from prometheus_flask_exporter import PrometheusMetrics

app = Flask(__name__)
metrics = PrometheusMetrics(app)  # adds /metrics

REDIS_HOST = os.getenv("REDIS_HOST", "localhost")
r = redis.Redis(host=REDIS_HOST, ...)

@app.route("/health")
def health():
    r.ping()  # confirms Redis is alive
    return jsonify({"status": "ok"}), 200

@app.route("/count", methods=["GET","POST"])
def count():
    if request.method == "POST":
        visits = r.incr("visit_count")  # atomic
        return jsonify({"visits": visits})
    visits = int(r.get("visit_count") or 0)
    return jsonify({"visits": visits})
WHY env var for REDIS_HOST?

Same code works everywhere. Local dev uses localhost, Docker Compose uses the service name redis, Kubernetes uses a ClusterIP. Zero code changes.

WHY r.incr() not r.get()+1?

INCR is a single atomic Redis command — thread-safe. GET then +1 then SET is a race condition under concurrent requests.

WHY PrometheusMetrics?

One line auto-instruments every route — request count, latency histograms, in-flight requests — all exposed at /metrics for Prometheus to scrape.

layer 2 — container

Multi-stage Dockerfile

Dockerfile
# ── STAGE 1: builder ────────────────────
FROM python:3.11-slim AS builder
WORKDIR /install
# Copy requirements FIRST (layer caching!)
COPY app/requirements.txt .
RUN pip install --prefix=/install/deps \
    --no-cache-dir -r requirements.txt

# ── STAGE 2: runtime ────────────────────
FROM python:3.11-slim AS runtime
# Non-root user — security best practice
RUN adduser --system appuser
COPY --from=builder /install/deps /usr/local
COPY app/ .
USER appuser
EXPOSE 5000
HEALTHCHECK --interval=30s ...
CMD ["gunicorn", "--workers", "2", \
     "--threads", "4", "main:app"]
WHY two stages?

Stage 1 has pip + build tools to compile packages. Stage 2 copies only the output — no pip, no compilers in production. Final image is ~120MB vs ~900MB.

WHY COPY requirements.txt FIRST?

Docker caches each layer. If requirements.txt hasn't changed, the pip install layer is reused — the slowest step is skipped on every code change.

WHY gunicorn not flask run?

Flask's dev server is single-threaded and not safe for production. Gunicorn runs 2 workers × 4 threads = 8 concurrent requests handled properly.

layer 3 — orchestration

Docker Compose

docker-compose.yml (excerpt)
services:
  flask-app:
    build: .
    ports: ["5000:5000"]
    env_file: [.env]
    environment:
      REDIS_HOST: redis  # ← service name!
    depends_on:
      redis: {condition: service_healthy}
    networks: [backend, monitoring]

  redis:
    image: redis:7-alpine
    command: redis-server --appendonly yes
    volumes: [redis_data:/data]
    networks: [backend]

volumes:
  redis_data:  # persists across restarts

networks:
  backend:    # app ↔ redis
  monitoring: # app ↔ prometheus
HOW does Flask find Redis?

Services on the same Docker network can reach each other by service name. REDIS_HOST=redis resolves to the Redis container's internal IP automatically via Docker's embedded DNS.

WHY named volume for Redis?

docker compose down destroys containers but named volumes survive. Your counter data persists. down -v is the "nuclear option" that deletes volumes too.

WHY two networks?

Principle of least privilege. Redis is on backend only — Prometheus can't reach it directly. The app bridges both networks because it needs to talk to both.

.github/workflows/ci-cd.yml (key parts)
on: { push: { branches: [main] } }

jobs:
  test:
    services:
      redis: { image: redis:7-alpine }  # real Redis in CI!
    steps:
      - flake8 app/ --max-line-length=120
      - pytest tests/ -v --cov=app

  build-and-push:
    needs: test  # ← only runs if tests pass
    if: github.ref == 'refs/heads/main'  # ← not on PRs
    steps:
      - docker/build-push-action
          tags: sha-${{ github.sha }}, latest
          cache-from: type=gha  # GitHub layer cache

sha-abc1234 tag = traceability. You always know exactly which commit is in production.

Secrets DOCKERHUB_USERNAME + DOCKERHUB_TOKEN stored in repo settings — never in code.
